# OrderControl v2.0

OrderControl - локальная веб-система складского учета с поддержкой нескольких складов, ролевым доступом, документами прихода и отпуска, инвентаризацией, ценами, PDF и журналом действий.

## Главное в версии 2.0

- Полное разделение товаров, остатков, документов и движений по складам.
- Глобальный переключатель рабочего склада для Руководителя.
- Жесткая привязка Менеджера и Продавца к назначенному складу.
- Отдельная номенклатура для каждого склада; одинаковый SKU разрешен на разных складах.
- Черновики отпуска резервируют товар и уменьшают доступный остаток.
- Документы прихода, отпуска и инвентаризации с изменяемыми названиями.
- Цены прихода и отпуска с разными правилами видимости.
- Формирование PDF по складским документам.
- Актуальная главная панель по выбранному складу.

## Склады

Каждый товар принадлежит одному складу. Документы, движения и остатки также хранятся в контексте конкретного склада.

- **Руководитель** выбирает активный склад в боковой панели и работает с его товарами и документами.
- **Менеджер** и **Продавец** не могут переключать склад и работают только со складом из своей учетной записи.
- Сервер проверяет принадлежность товара складу документа и отклоняет подмену товара через запрос или форму.
- При открытии документа Руководителем рабочий контекст переключается на склад этого документа.

## Роли и права

| Возможность | Продавец | Менеджер | Руководитель |
|---|---:|---:|---:|
| Просмотр своего склада | Да | Да | Да, с выбором склада |
| Создание черновиков документов | Да | Да | Да |
| Проведение документов | Нет | Да, на своем складе | Да |
| Изменение проведенных документов | Нет | Нет | Да |
| Создание товаров | Нет | Нет | Да |
| Редактирование товаров своего склада | Нет | Да | Да |
| Управление ценой отпуска | Нет | Да | Да |
| Просмотр цены прихода | Нет | Нет | Да |
| Создание пользователей | Нет | Только Продавцов своего склада | Да |
| Создание складов и назначение пользователей | Нет | Нет | Да |
| Просмотр полного журнала аудита | Нет | Нет | Да |

Технические значения ролей в базе: `viewer`, `manager`, `admin`.

## Товары и цены

Карточка товара содержит название, SKU, категорию, подкатегорию, единицу измерения, изображение, минимальный остаток, цену отпуска и активность.

- SKU уникален в пределах одного склада.
- Создавать товары может только Руководитель.
- Цена прихода указывается в позиции документа прихода, видна только Руководителю и не участвует в других разделах.
- Цена отпуска хранится в карточке товара, может изменяться Менеджером или Руководителем и видна всем ролям.
- В документе отпуска сохраняется цена позиции на момент оформления.
- Поиск товаров поддерживает название, SKU, категории и подкатегории.
- Список подкатегорий в фильтрах зависит от выбранной категории.

## Документы и остатки

Поддерживаются три типа документов:

- **Приход** - добавляет товар на склад после проведения.
- **Отпуск** - списывает товар после проведения и может быть связан с клиентом.
- **Инвентаризация** - приводит фактический остаток к указанному количеству.

Для нового документа автоматически создается номер вида `IN-...`, `OUT-...` или `INV-...`. Название можно изменить до сохранения и при разрешенном редактировании.

Статусы документов:

- `draft` - черновик;
- `posted` - проведен.

Черновик отпуска резервирует указанное количество. Доступный остаток рассчитывается как проведенный остаток минус резерв остальных черновиков, поэтому нельзя создать несколько черновиков на количество больше доступного.

Руководитель может исправлять проведенные документы. При сохранении система пересчитывает связанные движения склада.

## Основные возможности

- Главная панель: количество товаров, позиции не ниже минимума, дефицит и операции за текущий день.
- Каталог и табличный список товаров с фильтрами, поиском и изображениями.
- Клиенты и привязка клиента к документу отпуска.
- Категории и подкатегории.
- Документы прихода, отпуска и инвентаризации.
- Журнал движений с фильтрами по товару, клиенту, типу и датам.
- PDF для документов с учетом прав на цены прихода.
- Журнал аудита действий пользователей.
- Защита входа от перебора паролей и завершение неактивной сессии.
- Realtime-инвалидация страниц через polling.

## Запуск на Windows

1. Откройте PowerShell в папке проекта.

2. При первом запуске создайте виртуальное окружение и установите зависимости:

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
```

3. Проверьте настройки в `.env`.

4. Запустите приложение:

```powershell
python .\flask_app.py
```

5. Для текущей конфигурации откройте:

```text
http://127.0.0.1:4006
```

При первом запуске на пустой базе создается Руководитель:

- логин: `root`;
- пароль: `root`.

После первого входа пароль следует изменить в профиле.

## Настройки `.env`

```env
BLOCK_SEARCH_INDEXING=1
SQLITE_PATH=./data/ordercontrol.sqlite3
APP_PORT=4006
COOKIE_SECRET=replace-with-long-random-secret
APP_VERSION=v2.0
LOGS_MAX_ROWS=200000
TZ_OFFSET_HOURS=5
SESSION_IDLE_TIMEOUT_MINUTES=60
LOGIN_MAX_ATTEMPTS=5
LOGIN_BLOCK_MINUTES=10
APP_DEBUG=false
APP_CURRENCY=$
```

Основные параметры:

- `SQLITE_PATH` - путь к рабочей базе SQLite.
- `APP_PORT` - порт локального сервера.
- `COOKIE_SECRET` - секрет подписи сессий; в production должен быть уникальным и длинным.
- `APP_VERSION` - версия, отображаемая в интерфейсе.
- `TZ_OFFSET_HOURS` - смещение локального времени относительно UTC.
- `SESSION_IDLE_TIMEOUT_MINUTES` - завершение сессии при неактивности.
- `LOGIN_MAX_ATTEMPTS` и `LOGIN_BLOCK_MINUTES` - защита формы входа.
- `APP_CURRENCY` - обозначение валюты.
- `PRODUCT_UPLOAD_DIR` - необязательный путь для изображений товаров; по умолчанию `./static/uploads/products`.

## База данных

Приложение при запуске:

- создает SQLite-базу и таблицы, если их нет;
- применяет совместимые миграции схемы;
- назначает существующим данным склад при переходе со старой версии;
- включает внешние ключи и WAL;
- создает пользователя `root`, если таблица пользователей пуста.

Рабочая база по умолчанию: `data/ordercontrol.sqlite3`.

Перед ручным обновлением рекомендуется остановить приложение и сделать копию файла базы. В каталоге `data` уже может находиться резервная копия, созданная при переходе на многоскладскую модель.

## Основные маршруты

- `/` и `/admin` - главная панель активного склада.
- `/products` - товары активного склада.
- `/catalog` - визуальный каталог.
- `/clients` - клиенты.
- `/categories` и `/subcategories` - классификация товаров.
- `/stock/in` и `/stock/in/docs` - приход и список документов.
- `/stock/out` и `/stock/out/docs` - отпуск и список документов.
- `/stock/inventory` и `/stock/inventory/docs` - инвентаризация.
- `/stock/moves` - журнал движений.
- `/admins` - пользователи, роли и склады.
- `/profile` - профиль и смена пароля.
- `/logs` - аудит, доступный Руководителю.

## Развертывание

Для Passenger используется `passenger_wsgi.py`. Для production необходимо:

- задать надежный `COOKIE_SECRET`;
- оставить `APP_DEBUG=false`;
- включить HTTPS;
- регулярно копировать SQLite-базу;
- запускать приложение через WSGI-сервер, а не встроенный сервер Flask.
